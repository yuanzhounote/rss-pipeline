| 阶段          | 你（CTO）           | AI（开发工程师）          |
| ----------- | ---------------- | ------------------ |
| 产品方向        | ✅ 决定做什么、不做什么     | ❌ 不决策              |
| 系统架构        | ✅ 设计整体架构、模块边界    | ✅ 提出建议，按架构实现       |
| 技术选型        | ✅ 最终拍板           | ✅ 对比方案、分析优缺点       |
| Roadmap     | ✅ 制定版本计划         | ❌ 不决定优先级           |
| Issue 拆分    | ✅ 拆成可执行的小任务      | ✅ 可协助细化            |
| 详细设计        | ✅ 审核设计           | ✅ 输出设计文档           |
| 编码实现        | ❌ 很少亲自写          | ✅ 负责实现             |
| 单元测试        | ❌ 验收结果           | ✅ 编写测试、执行自测        |
| Code Review | ✅ 审核是否符合架构       | ✅ 自查、提出优化建议        |
| 部署脚本        | ❌ 确认流程           | ✅ 编写和维护            |
| 文档维护        | ✅ 架构文档、Roadmap   | ✅ README、API 文档、注释 |
| 最终验收        | ✅ 用真实场景测试、决定是否发布 | ❌ 不负责上线决策          |

**针对你现在的 RSS Pipeline 项目，我还有一个建议：**不要直接进入下一轮开发，而是先建立一套 `docs/` 文档体系（`ARCHITECTURE.md`、`ROADMAP.md`、`ISSUES/`、`PROMPTS/`）。这套文档以后会成为 AI 的长期上下文，你每次只需要给 AI 一个 Issue 和相关文档，它就能稳定地按你的架构开发，而不是每次都从头解释项目背景。长远来看，这会比单纯优化 Prompt 带来的收益更大。
