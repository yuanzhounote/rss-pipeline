## 1. 技术栈

- **运行时**：Cloudflare Workers + Cloudflare Queue + Cloudflare R2
- **数据库**：Supabase PostgreSQL
- **语言**：TypeScript
- **解析**：`@mozilla/readability` + `jsdom`
- **工具**：`wrangler` v4.105、`typescript` v6.0

依赖上有个隐患：

> `jsdom` 体积很大，且不是为 Workers 运行时设计的。后续如果要严格打包/减小体积，建议换成 `linkedom` 或 `cheerio` + 单独的 HTML 解析器。

## 2. 架构现状

项目已经按 **v2 方案** 的骨架搭好了：

```text
Telegram
    │
    ▼
Cloudflare Worker (API Gateway)
    │
    ▼
Supabase (pending)
    │
    ▼
Cloudflare Queue   ← 已配置 producer / consumer，但还没真正发消息
    │
    ▼
Extractor Worker
    │
    ▼
Supabase (ready)
    │
    ▼
```

## 3. 各文件职责

### `src/index.ts` — API Gateway

目前实现了 2 个路由：

- **`POST /webhook/telegram`**：提取消息里的 URL，写入 `articles` 表，状态为 `pending`。
- **`GET /rss.xml`**：查询 `status = 'ready'` 的文章，按 `published_at` 降序，实时生成 RSS XML。

**问题**：webhook 里 **直接同步调用 `extractArticle()`**，然后更新 `ready` 状态。  
也就是说 Extractor Worker 虽然存在，但**完全没有被使用**，Cloudflare Queue 也没有真正发消息。

### `extractor/src/extractor.ts` — Queue Consumer

- 暴露 `queue` 处理器。
- 从 Queue 消息里取 `articleId` + `sourceUrl`。
- 状态流转：`pending → extracting → uploading → ready`。
- 失败时更新 `status = failed` 并 `message.retry()`。
- 目前只有 `genericParser`，基于 Readability + JSDOM。
- 图片解析出来了（`images` 数组），但**还没有上传到 R2**。

### `sql/create_articles_table.sql`

表结构已经很完整，包含：

- `id`, `source_url`, `source_type`, `title`, `author`, `summary`, `cover`
- `content_html`, `content_md`
- `status`, `error`, `tags`
- `published_at`, `created_at`, `updated_at`

索引也到位：`status`、`published_at desc`、`source_url` 都有索引，还加了 `updated_at` 自动更新触发器。

### `wrangler.toml` / `extractor/wrangler.toml`

配置基本正确：

- API Worker 已声明 `[[queues.producers]]`。
- Extractor Worker 已声明 `[[queues.consumers]]` 和 `[[r2_buckets]]`。
- 环境变量都留了空，需要本地 `.env` 填充。

### `package.json`

缺少 `dev` / `deploy` 脚本。目前只有：

```json
"test": "echo \"Error: no test specified\" && exit 1"
```

建议补上：

```json
"dev": "wrangler dev",
"deploy": "wrangler deploy"
```

以及 Extractor 子目录的部署命令。

## 4. 当前关键缺口

| 缺口 | 影响 | 优先级 |
|---|---|---|
| **API Worker 没有真正发 Queue 消息** | Extractor 完全用不上，webhook 可能超时 | 高 |
| **`published_at` 未写入** | RSS 里 fallback 到 `created_at`，时间戳可能不准 | 中 |
| **图片没有处理** | RSS 里的图片仍是原站 URL，可能失效/防盗链 | 中 |
| **缺少部署脚本** | 手动输入 wrangler 命令容易出错 | 低 |
| **README 为空** | 别人/以后自己看不懂怎么跑 | 低 |
| **jsdom 体积问题** | 打包后可能体积过大，Worker 冷启动慢 | 低（可后续优化） |

## 5. 现在下一步最该做什么

如果按顺序推进，建议：

1. **把 `src/index.ts` 改成异步入队**（最高优先级）
   - webhook 收到 URL 后，写入 `pending` 行。
   - 通过 `env.QUEUE.send()` 发送 `{ articleId, sourceUrl }`。
   - 立即返回 “已加入处理队列”。
   - 删除 `src/index.ts` 里的同步 `extractArticle` 逻辑。

2. **补 `published_at` 写入**
   - 在 Extractor 解析完成后，把 `published_at` 一起写回数据库。

3. **处理图片**
   - 方案 A：Extractor 解析时上传图片到 R2，替换 HTML 里的图片 URL。
   - 方案 B：保留原图 URL，在 `GET /rss.xml` 或阅读时通过 Worker 图片代理懒缓存。

4. **补 package.json scripts 和 README**
