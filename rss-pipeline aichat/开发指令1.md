在现有 RSS Pipeline 基础上，**新增飞书机器人作为第二个入口**，不删除 Telegram。

## 1. 飞书机器人创建与配置

### 1.1 创建应用

1. 打开 [飞书开放平台](https://open.feishu.cn/app)
2. 创建企业自建应用
3. 进入「添加机器人」，启用机器人
4. 记录：
   - `App ID`
   - `App Secret`
   - `Encrypt Key`（如果启用加密）
   - `Verification Token`（旧版）或直接用事件订阅的签名验证

### 1.2 权限与事件订阅

需要开通的权限：
- `im:chat:readonly`
- `im:message:send`
- `im:message.group_msg`

事件订阅：
- 订阅事件 `im.message.receive_v1`
- 请求地址填：`https://<你的Worker>/webhook/feishu`

### 1.3 安装到群聊

进入应用「版本管理与发布」，创建版本并申请发布。通过后把机器人拉入目标群。

## 3. 签名验证逻辑

飞书 webhook 支持两种验证：

### 方案 A：Verification Token（简单，旧版）

事件请求体里有 `token` 字段，对比 `env.FEISHU_VERIFICATION_TOKEN` 是否一致。

### 方案 B：签名挑战（推荐，更安全）

请求头带：
- `X-Lark-Request-Timestamp`
- `X-Lark-Request-Nonce`
- `X-Lark-Signature`

签名算法：

```text
signature = HMAC_SHA256(FEISHU_APP_SECRET, timestamp + nonce + body)
```

对比请求头里的 `X-Lark-Signature`。

## 4. `src/index.ts` 新增路由

```typescript
if (request.method === 'POST' && url.pathname === '/webhook/feishu') {
  return handleFeishuWebhook(request, env);
}
```

### 4.1 `handleFeishuWebhook` 要做的事

1. 读取 body
2. 验证签名 / token
3. 处理飞书的 **URL 验证挑战**（首次配置事件订阅时，飞书会发一个带 `challenge` 字段的请求，必须原样返回）
4. 提取消息中的 URL
5. 检查 `source_url` 是否已存在（去重）
6. 插入 `articles`，`status = 'pending'`
7. 调用 `env.QUEUE.send({ articleId, sourceUrl })`
8. 更新 `status = 'queued'`
9. 回复飞书消息（可选）：`"已加入处理队列"`

### 4.2 飞书消息体结构

```json
{
  "schema": "2.0",
  "header": {
    "event_id": "...",
    "event_type": "im.message.receive_v1",
    "token": "...",
    "create_time": "..."
  },
  "event": {
    "message": {
      "message_type": "text",
      "content": "{\"text\":\"看看这篇文章 https://example.com/article\"}"
    },
    "sender": {
      "sender_id": { "open_id": "ou_xxx" }
    }
  }
}
```

注意：
- `content` 是 **JSON 字符串**，需要 `JSON.parse`
- 消息类型可能是 `text`、`post`、`image`，MVP 只处理 `text`
- URL 从 `text` 里用正则提取：`/https?:\/\/[^\s]+/`

## 5. 与 Telegram 入口复用

把 URL 提取、去重、入库、发 Queue 的逻辑抽成一个公共函数，比如：

```typescript
async function enqueueArticle(env: Env, sourceUrl: string, sourceType: string): Promise<{ id: number; status: string }>
```

Telegram 和飞书都调用它。

---

## 6. 回纷飞书消息

飞书回复需要调用 OpenAPI：

```text
POST https://open.feishu.cn/open-apis/im/v1/messages/{message_id}/reply
```

Header:
- `Authorization: Bearer <tenant_access_token>`

获取 `tenant_access_token`：

```text
POST https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal
Body: { "app_id": "cli_xxx", "app_secret": "xxx" }
```

MVP 阶段可以不回消息，先保证链路跑通。

## 7. 验收标准

1. 在飞书群里发一个普通网页链接
2. 1 秒后 `/health` 正常
3. Supabase 里出现一条 `source_type='feishu'`、`status='queued'` 的记录
4. Extractor 消费后变为 `ready`
5. `/rss.xml` 能返回这篇文章
6. 同一链接重复发送时，不插入重复记录

---

## 8. 风险提示

- 飞书 webhook 首次配置时会发送 `challenge` 验证请求，必须正确返回，否则事件订阅无法保存。
- 如果启用 Encrypt Key，需要先解密请求体再解析 JSON。
- 飞书机器人在群里必须 @机器人 才会收到消息，取决于权限配置。
