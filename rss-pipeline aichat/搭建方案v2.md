# RSS 信息管道搭建方案 v2

> 在 v1 基础上，根据架构反馈调整：
> 1. **Worker 不再负责重抓取**，只作为 API Gateway；抓取、解析放到 **Queue + Extractor** 异步执行。
> 2. **RSS 不再定时生成**，改为 `GET /rss.xml` 实时查询输出。
> 3. **数据库字段升级**，增加 `error`、`tags`、`published_at`、`summary` 等。
> 4. **解析器插件化**，新增来源只需新增 parser。
> 5. **图片优先懒缓存**，降低 R2 成本。

---

## 0. 总体架构

```text
Telegram
    │
    ▼
Cloudflare Worker（API Gateway / Webhook）
    │
    ▼
Supabase（articles.status = pending）
    │
    ▼
Cloudflare Queue
    │
    ▼
Extractor Worker
    ├─ Generic Parser（Readability）
    ├─ WeChat Parser
    ├─ Zhihu Parser
    ├─ Juejin Parser
    └─ ...
    │
    ▼
Article Normalize + Image Cache（R2）
    │
    ▼
Supabase（articles.status = ready）
    │
    ▼
GET /rss.xml（实时查询生成）
    │
    ▼
ReadYou / RSS 阅读器
```

## 1. 基础设施

| 组件 | 用途 | v2 配置 |
|---|---|---|
| **Cloudflare Worker** | 接收 webhook、输出 RSS | 1 个 API Gateway Worker |
| **Cloudflare Queue** | 异步任务队列 | 1 个 Queue，绑定 Extractor Worker |
| **Extractor** | 抓取、解析、规范化 | 第二阶段可用第二个 Worker 或轻量 VPS + 浏览器 |
| **Supabase** | 元数据 + 正文存储 | 1 张 `articles` 表 |
| **Cloudflare R2** | 图片缓存/持久化 | 1 个 bucket，按需回源 |
| **Telegram Bot** | 用户入口 | 第一阶段只接入 Telegram，飞书后续再加 |

> 先用 Telegram 跑通，因为 webhook 调试简单、文档完善。

## 2. 数据库表设计

```sql
create table articles (
  id bigint generated always as identity primary key,
  source_url text not null,
  source_type text,                    -- generic / wechat / zhihu / juejin / ...
  title text,
  author text,
  summary text,
  cover text,                          -- 封面图 URL
  content_html text,
  content_md text,
  status text not null default 'pending', -- pending / queued / extracting / uploading / ready / failed
  error text,                          -- 403 / timeout / captcha / parse_error / ...
  tags text[],                         -- 可空，后期用于分类
  published_at timestamptz,            -- 文章真实发布时间，用于 RSS <pubDate>
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

关键变化：
- 用 `published_at` 而不是 `created_at` 作为 RSS 时间戳。
- 增加 `error` 字段，方便排查失败原因。
- 增加 `summary` 和 `cover`，让 RSS 输出更完整。

## 3. Worker 端点设计

API Gateway Worker 只暴露 2 个核心路由：

### 1. `POST /webhook/telegram`
- 接收 Telegram 消息，提取 URL。
- 写入 `articles`，`status = 'pending'`。
- 发送一条消息到 Cloudflare Queue。
- 立即返回 `"已加入处理队列"`。

### 2. `GET /rss.xml`
- 查询 `status = 'ready'` 的文章。
- 按 `published_at DESC` 取前 100 条。
- 实时拼接 RSS 2.0 XML，返回 `Content-Type: application/xml`。
- 包含 `<guid>`、`<pubDate>`、`<content:encoded>` 等完整字段。

> 不需要 `/extract/:id` 路由，解析逻辑交给 Extractor 处理。

## 4. Extractor 抓取策略

Extractor 作为独立模块，负责从 Queue 消费任务并解析文章。

### 4.1 插件化解析器

每个解析器实现统一接口：

```ts
interface ArticleParser {
  name: string;
  canHandle(url: string): boolean;
  extract(url: string): Promise<{
    title: string;
    author?: string;
    summary?: string;
    cover?: string;
    content_html: string;
    content_md: string;
    images: string[];
    published_at?: Date;
  }>;
}
```

解析器目录：

```text
extractor/
├── parsers/
│   ├── generic.ts      # 基于 Readability 的通用兜底
│   ├── wechat.ts
│   ├── zhihu.ts
│   ├── juejin.ts
│   └── substack.ts
```

新增来源时，只需新增一个 parser 文件，无需改动主流程。

### 4.2 三级解析策略

| 层级 | 方式 | 适用场景 |
|---|---|---|
| 第一层 | 站点 API / 专用接口 | 微信、知乎等可获取 API 的情况 |
| 第二层 | Readability 通用解析 | 普通博客、新闻、落地页 |
| 第三层 | 无头浏览器（Browser Rendering / Puppeteer） | 强 JS 渲染、需要 Cookie 或登录的页面 |

> 先走 API，再走 Readability，最后上浏览器。成功率最高，成本也最低。

## 5. 状态机设计

```text
pending
   │
   ▼
queued
   │
   ▼
extracting
   │
   ▼
uploading        ← 图片异步处理阶段
   │
   ▼
ready
   │
   ▼
failed ───────────┘  （可重试回到 queued）
```

- **pending**：刚收到链接，等待入队。
- **queued**：已进入 Cloudflare Queue，等待 Extractor 消费。
- **extracting**：正在抓取/解析正文。
- **uploading**：正在处理图片（如异步上传 R2）。
- **ready**：解析完成，可进入 RSS。
- **failed**：失败，`error` 字段记录原因，支持重试。

## 6. 图片处理策略

推荐 **懒缓存（Lazy Cache）**，降低成本：

```text
网页原始图片 URL
        │
        ▼
RSS 中保留原始 URL
        │
        ▼
阅读器首次请求图片
        │
        ▼
Worker 图片代理回源
        │
        ▼
缓存到 R2 并返回给阅读器
```

如果目标是**永久存档**，可以在 Extractor 阶段异步上传所有图片到 R2 并替换 URL。MVP 阶段建议先懒缓存，避免为冷门文章支付大量 R2 写入费用。

## 8. 迭代路线

| 阶段 | 目标 |
|---|---|
| **MVP** | Telegram → Worker → Supabase → 动态 RSS；通用 Readability 解析；图片懒缓存 |
| **v2** | 引入 Cloudflare Queue + Extractor；完善状态机；失败重试；错误日志 |
| **v3** | 微信公众号、知乎、少数派等专用解析器；图片异步上传 R2；飞书机器人 |
| **v4** | 接入 ReadYou 或自建阅读器；支持分类、标签、全文搜索、阅读进度 |

## 9. 风险点与应对

| 风险 | 应对 |
|---|---|
| **反爬 / 被封** | 三级解析策略（API → Readability → Browser）；控制请求频率；轮换 User-Agent |
| **图片防盗链** | 图片代理回源时带 Referer、User-Agent |
| **RSS 输出不规范** | 确保 `<guid>`、`<pubDate>`、`<content:encoded>` 完整；`<guid>` 使用 `source_url` |
| **Worker 超时** | Webhook 只负责接收和入队，抓取交给 Queue；复杂任务设置重试 |
| **R2 成本** | 优先懒缓存；图片按需写入；避免大量预抓取 |
